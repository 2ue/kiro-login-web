# Kiro 登录工具速度分析

## 实测耗时分解（单账号成功案例）

基于日志 `dbc4a40b86f54051`：
- 09:36:33 开始
- 09:36:59 完成
- **总耗时：26 秒**

### 详细阶段：
1. **RegisterClient + StartDeviceAuthorization**: 09:36:33 → 09:36:35 (**2秒**)
2. **打开登录页**: 09:36:35 → 09:36:36 (**1秒**)
3. **页面加载到输入账号**: 09:36:36 → 09:42 (**6秒** - AWS页面加载+JS初始化)
4. **输入账号**: 09:36:42 → 09:36:44 (**2秒**)
5. **输入密码**: 09:36:44 → 09:36:48 (**4秒** - 包含metadata1等待)
6. **密码验证到授权页**: 09:36:48 → 09:36:55 (**7秒** - AWS后端验证)
7. **浏览器授权完成到换取token**: 09:36:55 → 09:36:57 (**2秒** - CreateToken轮询)
8. **扫描profileArn**: 09:36:57 → 09:36:59 (**2秒** - ListAvailableProfiles)

## 瓶颈识别

### 无法优化（AWS服务端）：
- AWS IAM页面加载（6秒）
- AWS后端密码验证（7秒）
- 总计：**13秒 = 50%耗时**

### 可优化部分：

#### 1. profileArn扫描策略 ⭐⭐⭐
**当前**：默认扫描 us-east-1 → eu-central-1 → ap-southeast-1
**问题**：即使首个区域找到profile，仍会继续扫所有区域
**优化**：找到即停止（已实现`exhaustive=False`参数）
**收益**：-2~4秒（多区域账号）

#### 2. metadata1等待超时 ⭐⭐
**当前**：`_wait_metadata1(timeout=30.0)`，实际AWS通常1-2秒生成
**问题**：并发高时CPU慢，JS生成token可能拖到5-8秒，但30秒过长
**优化**：降到15秒，0.4秒轮询改0.25秒
**收益**：不影响成功率，失败case提前返回

#### 3. 表单填充+验证 ⭐
**当前**：每个输入框填完后verify 4次，每次sleep 0.2秒 = 0.8秒
**优化**：改为verify 2次，sleep 0.1秒
**收益**：每个输入框-0.4秒，单账号约-1秒

#### 4. 密码页等待 ⭐
**当前**：`_wait_password_resolve(timeout=12.0)`，0.4秒轮询
**优化**：改0.3秒轮询
**收益**：响应快0.1秒左右

#### 5. 错峰随机延迟
**当前**：`time.sleep(random.uniform(0.05, 0.35))`
**优化**：改为`0.02~0.15`
**收益**：-0.15秒

#### 6. 图片/字体拦截
**当前**：已启用`route.abort()`
**状态**：✅ 已优化

## 预期收益

保守估计单账号可减少 **3-5秒**，从26秒降到**21-23秒**。

## 风险控制

- metadata1等待缩短可能在极高并发(>8)时误判，保留降级日志
- 表单verify减少可能在慢网络下填充不完整，保留最终verify
- 密码页等待缩短安全（AWS响应通常2-4秒）

## 不建议的优化

❌ 跳过cookie banner处理 - 可能卡住登录
❌ 取消page.set_default_timeout - 可能无限卡死
❌ headless模式 - 已是默认，无法再优化
